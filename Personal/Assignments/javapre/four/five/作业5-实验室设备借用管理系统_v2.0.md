# 《实验室设备借用管理系统》版本2.0：借用审批流程

## 技术栈

- 框架：Spring Boot 2.7.x
- 持久层：MyBatis 或 JPA（任选其一）
- 视图层：JSP + JSTL 或 Thymeleaf（任选）
- 数据库：MySQL
- 项目构建：Maven 或 Gradle（任选）

### 预置数据要求

- 复用版本1.0的数据库表结构（user/device/category三张表）
- 需在application.yml中正确配置数据库连接
- 保留已有的用户、设备数据

---

## 一、题目要求

### 1.1 背景说明

在版本1.0基础上，增加核心的设备借用申请与审批功能。用户可发起借用申请，管理员审批后完成借用，形成完整的审批流程。

### 1.2 功能需求

**（1）借用申请模块**

普通用户发起借用申请，填写信息：

| 字段 | 类型 | 说明 |
|------|------|------|
| apply_id | INT | 申请ID，主键自增 |
| user_id | INT | 申请人ID，外键 |
| device_id | INT | 设备ID，外键 |
| apply_date | DATETIME | 申请时间（由系统自动记录，不允许用户填写） |
| start_date | DATE | 计划借用开始日期 |
| end_date | DATE | 计划归还日期 |
| borrow_days | INT | 借用天数（由系统根据 end_date - start_date 自动计算，不允许客户端传入） |
| purpose_type | VARCHAR(20) | 借用用途类型：course（课程实验）/research（科研）/competition（竞赛）/other（其他） |
| purpose_note | VARCHAR(500) | 用途说明（当 purpose_type 为 other 时必填） |
| status | VARCHAR(20) | 状态：pending/approved/rejected/borrowed/returned/cancelled |
| audit_user_id | INT | 审批人ID，外键（审批操作完成后记录） |
| audit_time | DATETIME | 审批时间 |
| audit_note | VARCHAR(500) | 审批备注（驳回时必填） |
| actual_borrow_time | DATETIME | 实际借出时间 |
| actual_return_time | DATETIME | 实际归还时间 |
| return_note | VARCHAR(500) | 归还备注（如：设备完好/有轻微磨损） |

申请规则：
- 借用天数由后端根据 end_date - start_date 自动计算，接口不接受客户端传入 borrow_days
- 借用天数不能超过设备的最大借用天数（max_borrow_days）
- 同一设备同一时间段只能被一个用户借用（需实现时间冲突检测）
- purpose_type 为 other 时，purpose_note 为必填项

**借用申请输入校验（提交时在Service层逐项校验）**

| 校验字段 | 校验规则 |
|---------|---------|
| device_id | 不能为空；设备必须存在；设备状态必须为 available；设备未被退役（status != retired） |
| start_date | 不能为空；不能早于当前日期（不允许申请过去时间段） |
| end_date | 不能为空；必须晚于 start_date（不允许同一天借还，即 end_date > start_date） |
| 借用天数（后端计算） | 不能超过该设备的 max_borrow_days，超出时拒绝并提示允许的最大天数 |
| purpose_type | 不能为空；只能是 course/research/competition/other 之一 |
| purpose_note | purpose_type 为 other 时必填，长度不超过500字符 |
| 用户权限 | 申请人不能有 status=untreated 的违规记录（v3要求，v2可预留校验位） |
| 时间冲突 | 所选时间段内该设备不能有 status 为 pending/approved/borrowed 的已有申请 |

**（2）审批流程模块**

管理员对借用申请进行审批：

| 操作 | 前提状态 | 结果状态 | 说明 |
|------|---------|---------|------|
| 批准 | pending | approved | 同意借用申请 |
| 驳回 | pending | rejected | 拒绝申请，需填写驳回原因 |
| 确认借出 | approved | borrowed | 实际借出设备，记录实际借出时间 |
| 确认归还 | borrowed | returned | 归还设备，记录实际归还时间 |
| 取消申请 | pending | cancelled | 用户主动取消 |

审批操作输入校验：

| 操作 | 校验字段 | 校验规则 |
|------|---------|---------|
| 批准 | apply_id | 不能为空；申请必须存在；当前 status 必须为 pending |
| 批准 | （自动检查） | 设备当前 status 必须为 available；不能有时间段冲突的已批准/借出记录 |
| 驳回 | apply_id | 不能为空；当前 status 必须为 pending |
| 驳回 | audit_note | 必填，长度不超过500字符；不允许仅填空格 |
| 确认借出 | apply_id | 不能为空；当前 status 必须为 approved |
| 确认归还 | apply_id | 不能为空；当前 status 必须为 borrowed |
| 确认归还 | return_note | 必填，长度不超过500字符；不允许仅填空格 |
| 取消申请 | apply_id | 不能为空；当前 status 必须为 pending；只允许申请人本人取消（非管理员） |

其他审批要求：
- 确认借出时：自动更新设备状态为 borrowed，记录实际借出时间
- 确认归还时：自动更新设备状态为 available，记录实际归还时间
- 所有审批操作必须记录审批人ID和审批时间，审批人信息可追溯

**（3）管理员工作台**

管理员登录后，首页应展示以下数据看板：

| 看板项 | 说明 |
|--------|------|
| 待审批申请数 | 当前 status=pending 的申请数量 |
| 借出中设备数 | 当前 status=borrowed 的借用记录数 |
| 今日新增申请数 | 当天提交的申请数量 |
| 设备总数 | 各状态设备数量汇总 |

**（4）时间冲突检测**

借用申请时，自动检测该设备在申请时间段内是否已被其他用户借用或审批通过。

冲突判断规则：两个时间段存在重叠即为冲突
- 新申请：[start1, end1]
- 已有记录：[start2, end2]
- 冲突条件：start1 < end2 AND start2 < end1

需覆盖的场景：
- 完全包含（新申请时间段被已有记录完全包含）
- 包含已有（已有记录被新申请时间段完全包含）
- 局部重叠（两时间段部分重叠）
- 首尾相接（end1 = start2 或 start1 = end2，不算冲突）

**（5）借用记录查询**

用户可查看自己的借用记录（含历史记录），管理员可查看所有借用记录。记录展示包括：申请人、设备名称、借用时间、状态、操作。

**（6）我的申请页面**

用户登录后有独立的"我的申请"页面，展示当前用户的申请列表，包含：申请设备、申请时间、借用时间、当前状态、操作按钮（取消/查看详情）。

## 五、注意事项

- 复用版本1.0的数据库结构，只需新增borrow_apply表
- 设备状态与借用状态是两个独立概念，需要同步维护
- 时间冲突检测必须覆盖所有边界场景
- borrow_days 必须由后端计算，接口不接受客户端传值
- 审批、借出、归还等关键操作须在日志中记录操作人、操作时间和操作结果（可用控制台日志或数据库日志表，任选一种）
- 继承版本1.0的统一Result返回格式和全局异常处理机制
