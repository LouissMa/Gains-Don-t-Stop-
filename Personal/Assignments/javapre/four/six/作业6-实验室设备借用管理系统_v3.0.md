## 一、题目要求

### 1.1 背景说明

在版本2.0基础上，增加超时检测与违规记录功能。系统自动识别超时未还的设备并生成违规记录，违规记录将影响用户的借用权限，形成完整的信用约束机制。

### 1.2 功能需求

**（1）超时自动检测模块**

系统需自动检测当前借出设备中是否存在超时情况：

超时定义：当前时间 > 计划归还时间 且 借用状态仍为 borrowed

检测触发方式（任选其一实现）：
- 方式A：定时任务（使用@Scheduled注解，每小时/每天检测一次）
- 方式B：访问触发（用户访问借用记录页面时自动检测）
- 方式C：登录触发（用户登录时检测其借用的设备）

超时检测逻辑：
```
1. 查询所有 status='borrowed' 的借用记录
2. 对每条记录，比较当前时间与计划归还时间(end_date)
3. 若 current_time > end_date，标记为超时
4. 为超时记录生成违规记录（已有违规的不重复生成）
```

**（2）违规记录模块**

| 字段 | 类型 | 说明 |
|------|------|------|
| violation_id | INT | 违规ID，主键自增 |
| user_id | INT | 违规用户ID，外键 |
| device_id | INT | 违规设备ID，外键 |
| borrow_record_id | INT | 关联的借用记录ID，外键，唯一（一条借用记录只能对应一条违规） |
| violation_start_date | DATE | 违规开始日期（计划归还日期的次日） |
| violation_days | INT | 首次生成时的违规天数（快照值，不随时间变化） |
| status | VARCHAR(20) | 状态：untreated/processed |
| process_date | DATETIME | 处理时间 |
| process_user_id | INT | 处理人ID，外键（记录哪位管理员处理的） |
| process_note | VARCHAR(500) | 处理备注（必填） |

违规记录生成规则：
- 每条借用记录只生成一条违规记录（borrow_record_id 有唯一约束，重复检测时不重复生成）
- violation_start_date = 计划归还日期次日
- violation_days 字段在生成时记录快照值（首次检测时的超时天数），不随时间变化
- 展示违规详情时，当前实际超时天数需实时计算：current_date - 计划归还日期，而非直接读取 violation_days
- 违规记录状态默认为 untreated
- 处理违规时，process_note 为必填项，并记录处理人ID

**违规处理操作输入校验**

| 操作 | 校验字段 | 校验规则 |
|------|---------|---------|
| 处理违规 | violation_id | 不能为空；违规记录必须存在 |
| 处理违规 | status（当前） | 当前 status 必须为 untreated，不允许重复处理已 processed 的记录 |
| 处理违规 | process_note | 必填；不允许仅填空格；长度不超过500字符 |
| 处理违规 | process_user_id | 由系统从当前登录会话中获取，不允许客户端传入 |
| 设备使用率查询 | device_id | 不能为空；设备必须存在 |
| 设备使用率查询 | start_date | 不能为空；必须是合法日期格式 |
| 设备使用率查询 | end_date | 不能为空；必须晚于 start_date；查询区间不超过365天 |

**（3）违规记录管理模块**

管理员功能：
- 查看所有违规记录列表（分页）
- 按用户/设备/状态筛选
- 查看违规详情（关联的借用记录信息、当前实时超时天数）
- 处理违规（将untreated改为processed，需填写处理备注，记录处理人ID和处理时间）
- 违规统计：各用户违规次数、设备被超时次数
- 设备使用率统计：查询指定设备在某时间段内的借用天数占比（借用天数 / 时间段总天数），以表格或简单图表形式展示

用户功能：
- 查看自己的违规记录（展示当前实时超时天数，非快照值）
- 查看自己的违规次数统计

**（4）权限影响模块**

用户的违规记录影响其借用权限：

| 条件 | 权限影响 |
|------|---------|
| 有 untreated 违规记录 | 禁止发起新的借用申请 |
| 仅有 processed 违规记录 | 允许借用 |
| 无违规记录 | 允许借用 |

申请借用时的权限校验：
```
1. 查询用户是否有 status='untreated' 的违规记录
2. 若有，拒绝申请，返回提示："您有待处理的违规记录，无法申请借用设备"
3. 若无，正常进行时间冲突检测和借用申请流程
```

**业务联动规则（重要）**

当管理员对一条已有 untreated 违规记录的借用申请执行"确认归还"操作时，系统须满足以下规则之一（需在题目提交时说明采用了哪种方案）：

- 方案A：强制要求管理员先处理违规记录，才允许执行归还操作。归还时若存在未处理违规，拒绝操作并提示。
- 方案B：允许直接归还，归还后自动将关联的违规记录标记为 processed，系统填写处理备注为"设备已归还，系统自动处理"。

无论采用哪种方案，都需保证归还完成后设备状态正确变为 available。

**（5）违规通知与提醒**

- 用户登录后，若有 untreated 违规记录，页面顶部显示警告提示
- 超时设备列表单独展示，方便管理员追踪处理

- 违规处理操作须记录处理人ID，确保操作可追溯
